PROJECT 01 · EMBEDDED SW DEVELOPMENT

OTA를 위한 Bootloader 설계 및
SW Binary 위변조 감지

개인 프로젝트 · 개발 중심

Infineon AURIX TC234LP 기반 ECU에서 업데이트 실패와 Binary 변경 상황에도 기존 애플리케이션을 보호하도록 UDS 리프로그래밍, Backup·Restore, SHA-256 무결성 검사를 연결했습니다.

목표
불완전하거나 변경된 업데이트 이미지로의 부팅을 방지하고 정상 애플리케이션 복구 경로를 구성했습니다.
역할
개인 프로젝트로 메모리 영역과 진단 절차 설계, Embedded C 구현, Flash 제어, Trace32 디버깅을 전담했습니다.
구현
UDS 0x10·0x27·0x34·0x36·0x37, Application Backup·Restore, SHA-256 비교와 실패 정책을 구현했습니다.
검증
정상 다운로드, 전송 순서 오류, 무결성 불일치, 복구 경로와 양방향 Flash Write를 시나리오별로 확인했습니다.
결과
Memory Alignment Trap의 원인을 4바이트 정렬 조건으로 특정하고 수정 후 Application→Backup→Application 복구 흐름을 재검증했습니다.

KEY RESULTS

UDS 7개 서비스 흐름 Session·Security·Download·Reset 연계
Backup·Restore Primary와 Backup 영역 간 복구 경로
SHA-256 Application Binary 무결성 판정
Trap 원인 해결 Trace32 기반 4바이트 정렬 수정

시스템 구조와 리프로그래밍 흐름

Tester → CAN·ISO-TP → BswCom → BswDcm → RTE → ECU Abstraction → MCAL → PFlash·DFlash

Programming Session → SecurityAccess → Erase·Backup → RequestDownload(0x34) → TransferData(0x36) → RequestTransferExit(0x37) → SHA-256 확인 → ECUReset

핵심 구현

  • Application Primary를 Backup 영역으로 복사
  • Primary Erase 후 신규 SW Binary 기록
  • valid pattern과 SHA-256 결과로 실행 가능 여부 판정
  • 검사 실패 시 Backup을 Primary로 복구
  • TransferData의 Block Sequence와 재시도 실패 정책 처리

검증 범위

  • 정상 UDS 다운로드 및 TransferExit 완료
  • Block Sequence 불일치와 전송 중단 처리
  • 변조 Binary의 SHA-256 불일치 판정
  • Application 무효 시 Backup Restore 수행
  • Reset 이후 실행 경로와 저장 데이터 확인

대표 문제 해결 — Memory Alignment Error

증상 FlsLoader_Write 호출 중 Trap이 발생해 Backup·Restore가 중단됐습니다.
원인 분석 Trace32의 Trap 정보와 함수 인수를 추적해 source buffer의 정렬 조건을 점검했습니다.
수정 저장 버퍼를 uint32 배열로 변경하고 바이트 처리는 uint8*로 참조해 4바이트 정렬을 보장했습니다.
재검증 Application→Backup과 Backup→Application 양방향 Write·Read 및 SHA-256 판정을 다시 수행했습니다.

PROJECT 02 · VEHICLE SW VERIFICATION

CANoe/CAPL 기반 차량 ECU
IVS Black Box Validation

프로젝트 우수상 · 검증 중심

차량 ECU의 Fault Detection·Recovery·Clear 요구사양을 테스트 가능한 조건과 판정 기준으로 전환하고, CANoe/CAPL 기반 수동·자동화 시험으로 결함을 재현하고 문서화했습니다.

목표
Fault 상태 전이의 경계값, 선행 조건과 시간 요구사항을 정량 기준으로 검증했습니다.
역할
요구사양 분석, 테스트 케이스 설계, CANdb·CANoe 환경 구성, CAPL 자동화, Trace 분석과 결함 문서화를 수행했습니다.
구현
전제조건·입력·기대 결과·판정 기준을 CAPL 시나리오로 구현하고 Fresh Frame 기반 타이밍 측정 유틸리티를 구성했습니다.
검증
정적 검토와 동적 시험을 병행해 Detection·Recovery·Clear, 경계값, IGN Cycle과 메시지 Timing을 확인했습니다.
결과
정적 결함 4건과 동적 결함 11건을 식별하고 재현 조건·실제 결과·영향도·개선 방향을 정리해 프로젝트 우수상을 받았습니다.

KEY RESULTS

정적 결함 4건 Signal Length·Value·Description 불일치
동적 결함 11건 경계값·선행 조건·Recovery·Timing
CAPL 회귀 테스트 반복 가능한 자동화 시험 구성
프로젝트 우수상 검증 수행과 결함 분석 결과 인정

산출물

테스트 수행 흐름

요구사양 분석 → CANdb·CANoe·Panel 환경 구성 → 수동 시험 → CAPL 자동화 시험 → 정적·동적 검증 → 결함 분석 및 회귀 테스트

자동화 구현 범위

  • Detection·Recovery·Clear 테스트 시나리오
  • 경계값과 선행 조건 조합 입력
  • CAN Signal·System Variable 기반 판정
  • Fresh Frame 확인 후 타이머 시작 유틸리티

대표 검출 결과

  • 정적 결함 4건: Signal Length, Value, Description 불일치
  • 동적 결함 11건: 경계값, 선행 조건, Recovery, Timing 오류
  • Batt Percent 15% 경계값과 IGN 50 Cycle Off-by-One 확인
  • Steering Timing 요구사양 50±10ms 대비 약 1000±10ms 검출

대표 문제 해결 — Fresh Frame 동기화

증상 메시지 송신 방식 변경 후 이전 프레임을 기준으로 타이머가 시작돼 Detection 시점 판정이 흔들렸습니다.
원인 분석 CAN Trace Timestamp와 CAPL Flag 전환 시점을 비교해 최신 Reference Frame 수신 전 측정이 시작됨을 확인했습니다.
수정 waitBattReference·waitIGNReference에서 Fresh Frame을 확인한 뒤 1ms 타이머를 시작하도록 변경했습니다.
재검증 경계값·Timing 시나리오를 CAPL 회귀 테스트로 반복 실행해 동일 판정 기준이 유지되는지 확인했습니다.

실제 수행 환경 및 Test Result
